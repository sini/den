## What is a Stage

A **stage** is a named scope under `den.stages` that binds behavior to a point in Den's
resolution pipeline. Stages replaced the old `den.ctx` system with a cleaner model:
each stage declares what it **provides** (aspect fragments) and what it **includes**
(reusable behavior modules).

Den's four-concern model separates _what entities are_ (schema), _how they relate_
(policies), _where behavior binds_ (stages), and _how entities resolve_ (aspects).
Stages are the "where" -- they anchor aspect contributions to a specific pipeline step.

## Structure

```nix
den.stages.<name> = {
  provides.<aspect-name> = { host, ... }: <aspect-fragment>;
  includes = [ <provider> ... ];
};
```

- **`provides`** -- an attrset mapping names to functions that receive entity context
  and return aspect fragments. This is how a stage contributes configuration.
- **`includes`** -- a list of reusable providers (from `den.provides.*` or inline
  functions) that are mixed into this stage.

### Nested namespaces

`den.stages` is a recursive tree: any key that isn't detected as a leaf becomes
a namespace containing further stages. For example, `den.stages.flake-system`
contains sub-stages like `flake-os` and `flake-hm` for different output types.

<Aside type="caution">
Leaf detection relies on structural keys (`_`, `includes`, `provides`, `_module`).
If you only set freeform class keys -- e.g. `den.stages.foo.nixos.bar = "baz"` --
the node won't be recognized as a leaf and will be treated as a namespace.
Add `includes = [];` as a workaround to force leaf detection.
</Aside>

## Bare-function coercion

<Aside type="tip">
You can assign a bare function directly to a stage name. Den coerces it into
the canonical form automatically:

```nix
# This shorthand...
den.stages.host = { host }: host.aspect;

# ...becomes:
den.stages.host.provides.host = { host }: host.aspect;
```

The `provides` key is named after the stage itself.
</Aside>

## Attaching behavior to stages

To add behavior to a built-in stage like `host` or `user`, use `includes` --
this is the standard pattern for both batteries and user configuration:

```nix
den.stages.host.includes = [
  ({ host, ... }: { nixos.networking.hostName = "hello-${host.name}"; })
];
```

<Aside type="tip">
Use `includes` rather than `provides` when adding behavior to built-in stages.
The `provides` sub-key is primarily used when defining new stages from scratch.
</Aside>

Framework batteries use the same `den.stages.<name>.includes` pattern. For example,
the OS-class battery attaches forwarding providers to the `host` and `user` stages:

```nix
# modules/aspects/provides/os-class.nix (simplified)
den.stages.host.includes = [ host-os-fwd ];
den.stages.user.includes = [ user-os-fwd ];
```

Similarly, `import-tree` attaches directory-scanning providers:

```nix
den.stages.host.includes = [ (den.provides.import-tree.provides.host ./hosts) ];
den.stages.user.includes = [ (den.provides.import-tree.provides.user ./users) ];
```

The built-in stages (`host`, `user`, `wsl-host`, etc.) are defined by the framework
batteries. Your flake can define additional stages for custom pipeline steps.

## Built-in stages

| Stage | Context | Defined by |
|-------|---------|------------|
| `host` | `{ host }` | core |
| `user` | `{ host, user }` | core |
| `wsl-host` | `{ host }` | WSL battery |
| `flake-system` | flake outputs | core |
| `default` | -- | default aspects |

## Migrating from den.ctx

The old `den.ctx` namespace is preserved as a compatibility shim that forwards to
`den.stages` with deprecation warnings. The key differences:

- `den.ctx` was flat (no nested namespaces).
- `den.ctx.*.into` is gone -- transitions are now handled by [policies](/explanation/policies/).
- Stages don't have `__functor` -- they aren't callable.

To migrate, replace `den.ctx.<name>` with `den.stages.<name>` and move any `into`
definitions to `den.policies`.
