## `den.stages`

Type: `lazyAttrsOf stageTreeType`

A recursive tree of named scopes that bind behavior to pipeline stages.
Each leaf is a `stageSubmodule` (an aspect-like record without `into` or
`__functor`). Non-leaf nodes are nested namespaces (`lazyAttrsOf stageTreeType`).

Leaf detection uses key presence: any definition containing `_`, `includes`,
`provides`, or `_module` is treated as a leaf. Everything else is a namespace.

```nix
den.stages = {
  host.provides.host = { host }: host.aspect;   # leaf â€” has `provides`
  host.includes = [ den.provides.hostname ];     # same leaf, merged
  my-namespace.nested.leaf = {                   # namespace then leaf
    includes = [ someFn ];
  };
};
```

## Stage Leaf Options

Each leaf in the stage tree is a `stageSubmodule`. It shares the structural
options of an aspect but does not support `into` (transitions are handled by
[policies](/reference/policies/)) or `__functor` (stages are not callable).

### `provides` (alias: `_`)

Type: `attrsOf providerType`

Maps names to provider functions. When the pipeline resolves a stage, each
provider is called with the stage's context data and returns aspect fragments
that are merged into the output.

```nix
den.stages.host.provides.host = { host }: host.aspect;
den.stages.user.provides.user = { host, user }: user.aspect;
```

The `_` alias is shorthand for `provides`:

```nix
den.stages.host._.host = { host }: host.aspect;
```

### `includes`

Type: `listOf providerType`

A list of providers to include when resolving this stage. This is the
primary way to attach behavior to built-in stages like `host` and `user`.

```nix
den.stages.host.includes = [ den.provides.hostname ];
den.stages.user.includes = [ den.provides.mutual-provider ];
```

### `meta`

Type: `submodule`

Attached metadata. Notable sub-options:

- `meta.handleWith` -- resolution handlers for this stage's subtree.
- `meta.excludes` -- aspects to exclude from the subtree (sugar for `handleWith`).

### `name`

Type: `str`

The stage name, derived from its position in the tree.

### `description`

Type: `str`

Human-readable description of this stage.

## Function Sugar

A bare function assigned to a stage name is automatically coerced into a
provides entry keyed by that name:

```nix
# This:
den.stages.greet = { host }: { nixos.networking.hostName = host.name; };

# Becomes:
den.stages.greet.provides.greet = { host }: { nixos.networking.hostName = host.name; };
```

This makes single-provider stages concise to define.

## Built-in Stages

### `default`

Set via `den.stages.default` (or `den.default`). Behavior defined here appears
in every resolved output. The two forms coexist and are merged.

### `host`

Context: `{ host }`

Produced for each `den.hosts.<system>.<name>` entry. Providers:

- `provides.host` -- resolves to `host.aspect`.

### `user`

Context: `{ host, user }`

Produced for each user on a host. Providers:

- `provides.user` -- resolves to `user.aspect`.

### `home`

Context: `{ home }`

Produced for each `den.homes.<system>.<name>` entry. Providers:

- `provides.home` -- resolves to `home.aspect`.

### `hm-host`

Context: `{ host }`

Active when Home Manager integration is enabled on a host. Providers:

- `provides.hm-host` -- imports the Home Manager OS module.

### `hm-user`

Context: `{ host, user }`

Per-user Home Manager stage. Providers:

- `provides.hm-user` -- forwards `homeManager` class to `home-manager.users.<name>`.

### `wsl-host`

Context: `{ host }`

Active when WSL is enabled on a NixOS host. Providers:

- `provides.wsl-host` -- imports the WSL module.

### `flake-system`

Context: `{ system }`

Per-system stage for flake outputs. Providers:

- `provides.flake-os` -- OS configuration output.
- `provides.flake-hm` -- Home Manager configuration output.

<Aside type="caution" title="Deprecation: den.ctx">
`den.ctx` is a compatibility shim that forwards to `den.stages` with
deprecation warnings. The shim also maps `den.ctx.<name>.into` to
`den.stages.<name>.meta.into`. Transition support (the old `into` option)
is now handled by [policies](/reference/policies/). The shim will be
removed in a future release. See
[`modules/compat/ctx-shim.nix`](https://github.com/denful/den/blob/main/modules/compat/ctx-shim.nix).
</Aside>
